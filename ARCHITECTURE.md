# Архитектура Warcraft II на Godot 4.4

*Короткая часть для первого чтения: зачем проект устроен именно так и какие правила
нельзя ломать.*

Подробная рабочая часть вынесена отдельно: [ARCHITECTURE_DETAILS.md](ARCHITECTURE_DETAILS.md).

---

## Зачем нам архитектура

Мы пишем 2D-RTS под ОС «Аврора» на слабом железе. Требования по важности:

1. **Производительность** — стабильные 30 FPS на самом слабом устройстве.
2. **Малая связность** — модули не лезут друг другу во внутренности.
3. **Поддерживаемость** — код можно менять и тестировать по частям.
4. **Понятность** — студенты должны быстро понимать свой участок проекта.

Почти все решения ниже отвечают на один вопрос: что даст больше кадров в секунду и
при этом не превратит проект в кашу.

---

## Главная идея: логика отдельно от картинки

В наивном подходе юнит — это сцена Godot (`Node`), у которой внутри здоровье,
анимация и `_process()`, считающий урон. Для RTS на слабом устройстве это плохо:
тысячи узлов вызываются каждый кадр, а игровую логику трудно тестировать без сцены.

Поэтому мир делится на два слоя:

```text
        SIMULATION
  логика, правила, данные, такты
  единственный источник правды
              |
              v
        PRESENTATION / UI
  спрайты, анимации, камера, HUD, меню
  только показывает готовое состояние
```

**Simulation** не знает про сцены, спрайты, звук и файлы. Она считает игру как
обычный код. **Presentation** и **UI** читают подготовленные данные и рисуют их.
Если выключить картинку, игра всё равно должна считаться правильно.

> **Паттерн:** Model-View. Simulation — модель, Presentation/UI — представление.

## Юнит — это данные, а не Node

Юнит в симуляции — не объект Godot и не отдельный класс на каждый тип. Юнит — это
`EntityId` плюс набор компонентов в плотных типизированных массивах:

```gdscript
var positions: PackedVector2Array
var health: PackedInt32Array
var targets: PackedInt64Array
```

Такой подход важен для производительности:

- память лежит подряд, процессор лучше использует кэш;
- меньше объектов и мусора для сборщика;
- системы проходят только нужные сущности: движение обрабатывает тех, кто двигается,
  бой — тех, кто умеет атаковать.

Тип юнита определяется не наследованием `Knight -> Unit`, а набором компонентов и
ссылкой на данные контента. Рабочий = движение + здоровье + приказ + работа. Башня =
здание + здоровье + обзор + бой.

`EntityId` хранит индекс и поколение. Если юнит умер, его ячейку можно использовать
заново, но старая ссылка уже не попадёт случайно в новую сущность.

> **Паттерн:** Data-Oriented Design / Structure of Arrays.

## Время идёт тактами

Симуляция работает фиксированными шагами, например 20 тактов в секунду. Картинка
может рисоваться чаще и плавно интерполировать позиции между тактами.

Это даёт:

- **предсказуемость** — игра считается одинаково при разном FPS;
- **экономию** — тяжёлую логику не надо гонять каждый кадр;
- **защиту от зависаний** — если устройство не успевает, мы ограничиваем число
  догоняющих тактов за кадр.

> **Паттерн:** Fixed-timestep game loop + интерполяция представления.

## Модули и владельцы состояния

Главное правило: **у каждого состояния ровно один владелец**. Остальные модули
могут читать только подготовленные данные и только через разрешённые входы.

| Модуль | Ответственность |
| --- | --- |
| **App** | Запуск, экраны, жизненный цикл приложения. Не знает правил игры. |
| **Campaign** | Прогресс между миссиями, открытые миссии, результаты. |
| **Match** | Один игровой сеанс. Создаёт модули и соединяет их порты. |
| **Input** | Касания, выбор, жесты, создание команд. Не решает, допустимо ли действие. |
| **Simulation** | Вся правда о матче: юниты, ресурсы, бой, карта, AI, туман, такт. |
| **Scenario** | Цели, триггеры, волны, диалоги, обучение внутри матча. |
| **Presentation** | Игровой мир на экране: спрайты, камера, туман, миникарта, звук. |
| **UI** | Меню, HUD, панели, оверлеи. Показывает готовые данные. |
| **Services** | Сохранения, настройки, ресурсы, платформа, локализация, звук. |

> **Паттерн:** Single Source of Truth.

## Общение модулей: команды внутрь, данные наружу

Модули не читают чужие массивы и не правят чужое состояние напрямую. Внутрь
Simulation попадает только команда-намерение:

```text
Игрок / AI / Scenario
        |
        v
   GameCommand  ->  Simulation проверяет правила и меняет мир
                       |
                       +-> события для Scenario и Presentation
                       +-> dirty-буферы для Presentation
                       +-> ViewData для UI
```

Игрок, AI и сценарий используют одинаковые `GameCommand`. AI не двигает юнитов
в обход правил. Все проверки делает Simulation: хватает ли ресурсов, видна ли цель,
можно ли построить здание.

Наружу Simulation отдаёт только подготовленные данные через `game/simulation/ports/`.
UI не получает внутренние массивы, Presentation не читает хранилища напрямую.

> **Паттерн:** Command + Ports/Facade.

## Контент — это данные

Код описывает правила: как считается урон, как идёт добыча, как строится здание.
Конкретные юниты, здания, миссии, баланс и карты — это ресурсы (`.tres`) и каталоги.

```text
unit_definition.gd  -> схема: какие поля есть у юнита
units.tres          -> данные: footman, grunt, worker...
```

Новый юнит не требует нового класса вроде `Footman.gd`. Он добавляется в каталог,
получает визуальные данные и начинает работать через существующие системы.

> **Паттерн:** Data-Driven Design.

---

## Сохранение — снимок правды

Сохраняется только достоверное состояние матча: такт, RNG, компоненты, карта,
стороны, приказы, отложенные команды.

Не сохраняется то, что можно восстановить: спрайты, кэш путей, временные буферы,
визуальные эффекты. После загрузки они строятся заново.

> **Паттерн:** Snapshot / Memento.

## Минимальная карта проекта

```text
warcraft-ii/
├── app/              запуск, экраны, жизненный цикл
├── game/
│   ├── campaign/     прогресс между миссиями
│   ├── match/        сборка одного матча
│   ├── input/        жесты, выбор, создание команд
│   ├── simulation/   вся игровая логика и состояние
│   ├── scenario/     миссии, сюжет, обучение
│   └── presentation/ отображение игрового мира
├── ui/               меню, HUD, панели, оверлеи
├── services/         сохранения, настройки, ресурсы, платформа
├── content/          схемы, каталоги, баланс, карты, ассеты
├── tests/            unit / integration / performance
├── debug/            отладочные оверлеи
├── tools/            инструменты разработки и контента
└── docs/             подробные спецификации
```

Подробная структура папок и практический справочник лежат в
[ARCHITECTURE_DETAILS.md](ARCHITECTURE_DETAILS.md).

## Правила, которые держат проект

- **У каждого состояния один владелец.**
- **Юнит — данные в массивах, не `Node` с игровой логикой.**
- **Мир меняется только через команды.**
- **Simulation не знает про сцены, звук, файлы и платформу.**
- **Presentation и UI ничего не решают, только показывают.**
- **Новый контент — это `.tres` и каталоги, а не новый класс.**
- **Сначала замер на «Авроре», потом оптимизация.**
