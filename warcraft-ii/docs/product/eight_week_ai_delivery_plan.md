# 8-недельный план разработки для AI-assisted команды

Документ отвечает на вопрос, что делать в первую очередь, если за 8 недель нужно
собрать демонстрируемый порт Warcraft II для ОС Аврора, а весь код
пишется ИИ под контролем команды.

## Исходные условия

- Команда: 4 программиста-оператора ИИ, 1 дизайнер, 1 тестировщик.
- Стек старта: Godot 4.4, GDScript, `.tres`/каталоги данных, `.gdshader` для графики.
- Цель: gameplay-equivalent порт Warcraft II для ОС Аврора; новая визуальная оболочка не меняет основные механики, лор, сюжетную линию и миссии.
- Референс механик: `docs/gameplay/mechanics_matrix.md` и локальный Wargus как основной технический справочник.
- Точность: не byte-perfect реконструкция; отдельные технические отличия допустимы из-за мобильной платформы и ограничений проекта.
- Ограничение: не копировать GPL-код Wargus и оригинальные ассеты Warcraft II без прав.
- Формат результата: каждую пятницу есть запускаемое демо, отчет спринта и реальные проверки.

## Что делать в первую очередь

В первую очередь нужно получить не красивую картинку и не полный список механик, а
повторяемый технический путь:

1. Проект открывается в Godot 4.4 и запускает main scene.
2. Есть `Match` как composition root.
3. Есть `SimulationRunner` с фиксированным тиком.
4. Игрок, AI и сценарий могут отправлять только `GameCommand`.
5. `Simulation` меняет состояние, а `Presentation/UI` только читают подготовленные данные.
6. Есть минимальный smoke test и отчет спринта.

Это обязательно, потому что без такого каркаса ИИ быстро начнет писать удобный, но
опасный код: логику в UI, состояние в Node, прямые вызовы между слоями и одноразовые
скрипты, которые сложно тестировать.

## Связь с системой оценивания

План учитывает [`grading_rules_2026.md`](../evaluation/grading_rules_2026.md), но его нужно читать как инженерный план для
набора баллов, а не как гарантию максимальной оценки. Максимизация баллов зависит
не только от объема кода, но и от того, можно ли каждую пятницу открыть ссылки,
запустить демо и увидеть именно то, что команда заявила.

### Главный стоппер: 0 баллов за спринт

По критериям спринт получает 0, если нет доступных ссылок, проект не запускается
или демо не соответствует репозиторию. Поэтому каждый спринт обязан сначала
закрывать не новую механику, а воспроизводимость.

| Требование оценки | Как закрывает план | Что проверять каждую пятницу |
| --- | --- | --- |
| Ссылка на проект и материалы открывается | Все артефакты держать в GitVerse; отчеты в `docs/sprints/`. | Открыть ссылки из README и sprint report. |
| Приложение запускается и функционирует | Sprint 01 начинается с main scene, App/Menu/Match и smoke test. | Чистый запуск в Godot 4.4 по README. |
| Демо соответствует коду | Каждый спринт имеет Definition of Done и test/manual scenario. | Не показывать на демо то, чего нет в репозитории. |
| Материалы только в GitVerse | План требует README, reports, assets/mock assets, docs в репозитории. | Нет внешних файлов, презентаций и ассетов вне GitVerse. |

### Технические 5 баллов

В [`grading_rules_2026.md`](../evaluation/grading_rules_2026.md) техническая часть состоит из разработки и тестирования. Для
AI-assisted и классической модели формулировки отличаются, но практический вывод
одинаковый: нужна чистая кодовая база, единый стиль, понятная архитектура и тесты.

| Критерий | Максимум | Как план максимизирует |
| --- | ---: | --- |
| README/Wiki, Git-репозиторий, актуальная документация | 1 или часть 0-3 | README и `docs/README.md` ведут к архитектуре, USM, тестам, отчетам и плану. |
| Консистентность и общий архитектурный стиль | 1 или часть 0-3 | Simulation как источник правды, `GameCommand`, fixed tick, запрет логики в UI. |
| Качество кода / Code Smells / качество AI-кодовой базы | 1 или часть 0-3 | Задачи дробятся по слоям; у каждой есть владелец состояния, тест и DoD. |
| Тест-план Sprint 01 | 2 | Sprint 01 не гонится за механиками, а закрывает запуск, test strategy и smoke path. |
| Тест-кейсы Sprint 02-08 | 2 | В каждом спринте есть ручные/unit/integration/performance проверки по реализованным фичам. |

Для максимума технических баллов нельзя принимать задачу как готовую, если:

- нет тест-кейса или ручного сценария;
- README/отчет не описывает, как повторить демо;
- новая механика не связана с `mechanics_matrix.md`;
- UI, Presentation, Scenario или Input напрямую меняют состояние Simulation;
- в отчете написано больше, чем реально запускается.

### Продуктовые 5 баллов

Продуктовая часть оценивает питч, UX/UI и реализацию по User Story Map. Поэтому в
плане каждая неделя должна показывать пользовательский прогресс, а не только
внутренние модули.

| Критерий | Максимум | Как план максимизирует |
| --- | ---: | --- |
| Питч проекта | 2 | Каждую неделю фиксировать: для кого порт, что теперь можно сделать, какие Warcraft II-механики воспроизведены. |
| UX/UI дизайн | 2 | Дизайнер с Sprint 01 ведет новый стиль, HUD, читаемые иконки, экран результата, без чужих ассетов. |
| User Story Map | 1 | Каждая демо-фича привязана к USM и имеет проверяемую user story. |

Минимальный продуктовый артефакт каждого спринта:

1. 3-5 предложений для питча текущего демо.
2. Скриншот или макет измененного UI/игрового экрана в GitVerse.
3. Список user stories, которые реально работают.
4. Один короткий демо-сценарий: что нажать и какой результат увидеть.

### Почему Sprint 08 особенно важен

По правилам крайний спринт удваивается. Поэтому Sprint 08 в плане не добавляет
крупные новые системы, а фокусируется на запуске, регрессиях, производительности,
визуальной полировке и документации. Это дает больше ожидаемых баллов, чем попытка
в последний момент добавить магию, море, воздух или Lua-моды.

## Почему не начинать с Lua, C++ и полной кампании

Lua, C++/GDExtension, сложный AI, море, воздух, магия, полноценные кампании и
импорт всех данных из Wargus - это полезные направления, но не первый шаг.

Причина простая: пока нет стабильного `GameCommand`, снимка Simulation, тестов и
рабочего вертикального среза, любое расширение увеличивает хаос. Для команды, где
код пишет ИИ, это особенно рискованно: ИИ хорошо генерирует локальный код, но без
жестких границ легко нарушает архитектуру.

## Шкала оценки задач

Оценка "промптов" - это не количество сообщений в чате. Один промпт здесь означает
один управляемый цикл работы ИИ:

```text
контекст -> точная задача -> код -> тест/проверка -> правка -> обновление документации
```

| Оценка | Промпты | Что значит |
| --- | ---: | --- |
| XS | 1-2 | Маленькая правка в одном модуле. |
| S | 3-5 | Узкая задача с тестом или простой UI. |
| M | 6-10 | Несколько файлов, нужен контроль архитектурных границ. |
| L | 11-18 | Несколько подсистем, высокий риск регрессий. |
| XL | 19+ | Опасная задача для ИИ без тимлида, нужна декомпозиция. |

| Риск багов | Вероятность | Типичные причины |
| --- | --- | --- |
| Низкий | 10-25% | Локальная задача, мало связей. |
| Средний | 25-50% | Есть интеграция между слоями. |
| Высокий | 50-70% | Затрагивает Simulation, сохранение, навигацию, бой. |
| Критический | 70%+ | Нужен пересмотр архитектуры или много скрытых правил. |

| Сложность для ИИ без тимлида | Значение |
| ---: | --- |
| 1 | Можно дать новичку с готовым README и тестом. |
| 2 | Нужен короткий code review. |
| 3 | Нужен владелец модуля и проверка архитектурных границ. |
| 4 | Нельзя отдавать неопытному без пошагового плана и ревью. |
| 5 | Нельзя делать без тимлида; сначала дробить задачу. |

## Роли команды

Так как код пишет ИИ, программисты фактически работают как постановщики задач,
ревьюеры и интеграторы.

| Роль | Основная зона |
| --- | --- |
| Программист 1 | `game/simulation`: команды, storage, systems, deterministic tests. |
| Программист 2 | `game/presentation` и `game/input`: камера, выбор, views, пулы, интерполяция. |
| Программист 3 | `ui`, `scenario`, `campaign`: меню, HUD, цели, result screen. |
| Программист 4 | `content`, `services`, `tools`, тестовая инфраструктура, save/load. |
| Дизайнер | Новый визуальный стиль, UI layout, иконки, mock assets, readable RTS screen. |
| Тестировщик | Smoke test, ручные тест-кейсы, регрессии, проверка отчетов и known issues. |

Минимум один программист каждую неделю должен выполнять роль архитектурного
ревьюера: проверять, что ИИ не перенес игровую логику в UI/Presentation и не
обошел `GameCommand`.

## Что сознательно не входит в 8 недель

Эти задачи не нужно брать в первые 8 недель, если только они не стали блокером
демо. Они слишком дорогие для AI-first команды без зрелой кодовой базы.

| Задача | Почему не сейчас |
| --- | --- |
| Lua runtime для кампаний и модов | Нужны sandbox, API-контракты, сериализация и тесты. |
| C++/GDExtension ядро | Сначала нужны профилирование и доказанная необходимость. |
| Полный импорт всех Wargus Lua scripts | Высокий риск юридической и архитектурной путаницы. |
| Морские и воздушные юниты | Усложняют pathfinding, target rules и карту. |
| Магия, upgrades, autocast | Требуют зрелых ability/status systems. |
| Полная кампания | Достаточно одной демонстрационной миссии. |
| Multiplayer/network sync | Требует детерминизма и отдельной архитектуры. |
| Редактор карт | Не нужен для DemoFest MVP. |

## Правила постановки задач ИИ

Каждая задача для ИИ должна включать:

1. Файл/модуль, который можно менять.
2. Файлы, которые нельзя трогать.
3. Ссылку на `mechanics_matrix.md`, если задача добавляет Warcraft II-механику.
4. Ожидаемый тест или ручной сценарий.
5. Архитектурное правило: кто владеет состоянием.
6. Запрет на прямой доступ UI/Presentation к Simulation storage.

Плохая постановка:

```text
Сделай строительство как в Warcraft II.
```

Хорошая постановка:

```text
В `game/simulation` реализуй MVP команды `build` для Farm:
- вход только через `GameCommand`;
- ресурсы списывает `ConstructionSystem`;
- placement проверяет occupancy grid;
- UI не читает storage напрямую;
- добавь unit-тест: нельзя строить на занятой клетке;
- обнови Sprint 05 report и укажи mechanics_matrix IDs.
```

## Минимальный Definition of Done для любой задачи

- Код находится в правильном модуле.
- Simulation не зависит от Godot-сцен, UI, Presentation, звука или файловой системы.
- Игровое действие игрока, AI или сценария идет через `GameCommand`.
- Есть unit/integration/manual test.
- Документация или sprint report обновлены, если изменилось поведение.
- Известные ограничения записаны явно.

## Главный объективный вывод

Команде нужно начинать с каркаса запуска, fixed tick, `GameCommand` и тестируемой
Simulation. Это менее эффектно, чем сразу рисовать полный Warcraft II, но это
единственный реалистичный путь для проекта, где код генерирует ИИ.

Если сначала сделать UI, визуалы и много разрозненных скриптов, через 3-4 недели
команда получит демо, которое трудно расширять и невозможно надежно тестировать.
Если сначала сделать авторитетную Simulation и узкие порты, то каждую следующую
механику можно добавлять как проверяемую систему, а не как новый хаос.
