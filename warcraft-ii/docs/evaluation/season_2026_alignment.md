# Соответствие системе оценки сезона 2026

Документ переводит требования из `оценка работы.txt` в рабочий чеклист проекта. Его задача - не хранить полный текст правил, а снижать риск потери баллов из-за организационных ошибок.

## Календарные ориентиры

- Сезон состоит из 8 недельных спринтов.
- Первый спринт стартует 6 июля 2026.
- Восьмой спринт стартует 24 августа 2026.
- DemoFest запланирован на 12 сентября 2026.
- Промежуточное онлайн-демо проходит по пятницам с 18:00 до 20:00.
- Проверка артефактов начинается по пятницам в 21:00 GMT+6.

## Риски, которые дают 0 за спринт

Перед сдачей нужно проверить:

- ссылка на репозиторий открывается;
- ссылка на демо, билд или инструкцию запуска открывается;
- проект запускается по инструкции из README;
- все материалы проекта лежат в GitVerse;
- отчет спринта лежит в `docs/sprints/`;
- реализованное на демо соответствует коду в репозитории;
- в отчете нет обещаний, которые нельзя повторить запуском проекта.

## Техническая часть

| Критерий | Как закрываем в проекте | Где фиксируется |
| --- | --- | --- |
| README/Wiki актуальны | Корневой README ведет к запуску, архитектуре, тестам и отчетам. | `README.md`, `warcraft-ii/README.md`, `docs/README.md` |
| Общий архитектурный стиль | Используем Simulation как источник правды, команды внутрь, данные наружу. | `ARCHITECTURE.md`, `ARCHITECTURE_DETAILS.md`, `AGENTS.md` |
| Качество кода | Соблюдаем владельцев состояния, не смешиваем UI и игровую логику, добавляем тесты. | README модулей, `AGENTS.md`, `tests/` |
| AI-assisted качество кодовой базы | Даем ИИ явные инструкции и запреты, чтобы оценщик видел намеренную структуру. | `AGENTS.md` |
| Тест-план первого спринта | Описываем цели, подходы, bug-fix flow, сценарии и готовность к тестированию. | `docs/testing/test_strategy.md` |
| Тест-кейсы спринтов 2-8 | Для каждой реализованной фичи добавляем структурированные тест-кейсы. | `docs/sprints/sprint_XX_report.md`, `docs/testing/test_strategy.md` |

## Чеклист перед пятницей

1. Открыть проект в Godot 4.4.
2. Запустить основной сценарий демо.
3. Проверить, что нет критических ошибок в Output.
4. Пройти smoke test из `docs/testing/test_strategy.md`.
5. Обновить отчет текущего спринта.
6. В отчете указать, какие тесты запускались.
7. Проверить, что README ведет к актуальному отчету.
8. Проверить, что все ссылки доступны из GitVerse.
9. Убедиться, что не используются внешние артефакты вне GitVerse.
10. Зафиксировать известные ограничения честно и кратко.

## Формула спринта

Каждый спринт должен отвечать на четыре вопроса:

- Что пользователь теперь может сделать?
- Как это запустить?
- Как это проверено?
- Какие ограничения остаются?

Если на эти вопросы нельзя ответить из README и отчета спринта, документацию нужно обновить до сдачи.
