# Workflow разработки

Документ описывает, как команда берет задачу, ставит ее ИИ, проверяет результат и
готовит изменение к сдаче спринта. Он нужен, чтобы четыре программиста-оператора ИИ
работали в одном стиле и не ломали архитектурные границы.

## Перед началом задачи

1. Прочитать `AGENTS.md`.
2. Прочитать `docs/architecture/architecture.md`.
3. Прочитать `docs/architecture/architecture_details.md`.
4. Прочитать README модуля, который меняется.
5. Если задача добавляет механику Warcraft II, найти строку в `docs/gameplay/mechanics_matrix.md`.
6. Если задача входит в спринт, открыть актуальный отчет в `docs/sprints/`.

## Жизненный цикл задачи

| Шаг | Что делает исполнитель | Что должно остаться в репозитории |
| --- | --- | --- |
| 1. Уточнить границы | Определить модуль-владелец состояния и запрещенные слои. | Заполненная задача по `sprint_task_template.md` или issue. |
| 2. Сформулировать промпт | Дать ИИ контекст, DoD, тест и запреты. | Промпт можно сохранить в задаче, если он важен для ревью. |
| 3. Получить изменение | Проверить, что ИИ не вышел за границы модуля. | Код/документы только в ожидаемых местах. |
| 4. Проверить | Запустить доступные тесты или ручной сценарий. | Результат проверки в sprint report. |
| 5. Обновить документы | Обновить README, тест-кейсы, матрицу механик или отчет спринта. | Документы совпадают с реальным поведением. |
| 6. Review | Проверить архитектуру, ссылки, known issues и запуск демо. | Исправленные замечания или явные ограничения. |

## Минимальный review checklist

- Изменение лежит в правильном модуле.
- Игровая логика не попала в UI, Presentation, Godot Node или сцену.
- Действия игрока, AI и сценария идут через `WarcraftCommand`.
- Warcraft Runtime не зависит от файловой системы, звука, UI и Presentation.
- Новая Warcraft II-механика имеет ссылку на `mechanics_matrix.md`.
- Есть unit/integration/manual проверка.
- Sprint report говорит только о том, что реально запускается.
- Локальные Markdown-ссылки не сломаны.

## Правило пятницы

Перед пятничной проверкой команда сначала закрывает воспроизводимость:

1. Чистый запуск проекта по README.
2. Проход демо-сценария текущего спринта.
3. Проверка README и sprint report.
4. Фиксация тестов, которые реально запускались.
5. Фиксация known issues.

Новые фичи после этого можно добавлять только если они не рискуют сломать демо.
