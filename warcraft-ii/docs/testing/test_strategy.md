# Стратегия тестирования

Документ закрывает тест-план первого спринта и задает формат тест-кейсов для спринтов 2-8. Он должен обновляться вместе с реальным функционалом.

## Цели

- Проверять, что проект запускается и демо можно повторить по README.
- Проверять RTS-логику отдельно от UI и сцен.
- Ловить регрессии в экономике, командах, движении, бое, тумане, сценариях и сохранениях.
- Подтверждать, что реализованные механики соответствуют `docs/gameplay/mechanics_matrix.md`.
- Фиксировать ручные тест-кейсы так, чтобы их мог пройти любой проверяющий.

## Текущее состояние тестов

В `tests/` заведены папки `unit`, `integration` и `performance`, а также файлы будущих проверок. На старте часть тестовых файлов может быть пустой. До оценки спринта каждый заявленный тест должен получить реальное тело, команду запуска или ручной сценарий проверки.

## Уровни тестирования

| Уровень | Что проверяет | Где хранится |
| --- | --- | --- |
| Unit | Отдельные системы Warcraft Runtime, сервисы, схемы данных, чистые алгоритмы. | `tests/unit/` |
| Integration | Потоки между Match, Warcraft Runtime, Scenario, Input, UI ports, save/load. | `tests/integration/` |
| Performance | FPS-бюджеты, память, массовое движение, бой, pathfinding, видимость. | `tests/performance/` |
| Manual smoke | Запуск проекта и основной пользовательский путь перед демо. | Отчет спринта |
| Content validation | Целостность каталогов, карт, миссий, ссылок на ассеты. | `tools/`, отчет спринта |

## Приоритеты

| Приоритет | Область | Почему важно |
| --- | --- | --- |
| P0 | Запуск проекта, основной демо-путь, критические ошибки | Без этого спринт может получить 0. |
| P0 | Warcraft Runtime-команды и состояние матча | Это ядро RTS и источник правды. |
| P1 | Экономика, строительство, производство, бой, движение | Это основные Warcraft II-механики. |
| P1 | Сценарии миссий, цели, победа/поражение | Это показывает продуктовую ценность демо. |
| P1 | Сохранение/загрузка | Это высокий риск регрессий и потери состояния. |
| P2 | Асинхронная загрузка ассетов и cache | Нужна для плавного запуска/матча, но не должна менять gameplay state. |
| P2 | Отладочные оверлеи, вспомогательные инструменты | Полезно, но не должно блокировать демо. |

## Тест-план первого спринта

Первый спринт оценивает готовность к тестированию, поэтому должны быть подготовлены:

- этот документ;
- список smoke-проверок;
- шаблон тест-кейса;
- bug-fix workflow;
- первичный список рисков;
- связь тестов с User Story Map и `mechanics_matrix.md`.

| ID | Сценарий | Приоритет | Ожидаемый результат |

## Smoke test перед каждым демо

Перед пятничной сдачей пройти минимум:

1. Открыть `warcraft-ii/project.godot` в Godot 4.4.
2. Запустить main scene или `res://app/app.tscn`.
3. Дойти до демонстрируемого игрового сценария.
4. Выполнить одну основную user story текущего спринта.
5. Проверить, что нет критических ошибок в Output.
6. Проверить, что README и отчет спринта описывают тот же путь.
7. Если спринт затрагивает визуальные ассеты, проверить, что нет скрытой
   синхронной загрузки тяжелых ресурсов во время матча и fallback не ломает UI.
8. Если используются `original_placeholder`, проверить manifest entry,
   replacement target и отсутствие прямых ссылок из Warcraft Runtime.
9. Зафиксировать результат в отчете спринта.

## Проверки async asset loading

Когда реализуется `services/assets`, добавить отдельные проверки:

- unit: повторный запрос одного asset id не создает вторую загрузку;
- unit: `AssetLoader` не вызывает получение threaded resource до статуса loaded;
- integration: переход в матч запускает preload нужного mission/race/tileset
  пакета через manifest;
- integration: Presentation/UI получают fallback для еще не готового ресурса и
  заменяют его после ready без изменения Warcraft Runtime;
- performance/manual: запуск демо фиксирует время preload, отсутствие заметных
  фризов и список failed/missing assets.
- content/manual: каждый `original_placeholder` имеет запись в
  `content/assets/placeholder_manifest.json` и план замены;
- integration/manual: замена placeholder на `replacement_ready` меняет только
  visual/audio resource, но не save snapshot и не Warcraft Runtime state.

## Формат тест-кейса

```md
### TC-ID: Название

- User story:
- Механика из mechanics_matrix.md:
- Приоритет: P0/P1/P2
- Тип: manual/unit/integration/performance
- Предусловия:
- Шаги:
- Ожидаемый результат:
- Фактический результат:
- Статус: pass/fail/blocked/not run
- Связанный баг или ограничение:
```

## Bug-fix workflow

1. Зафиксировать симптом: что произошло, где, на каком шаге.
2. Указать ожидаемый результат.
3. Определить слой: Warcraft Runtime, Scenario, Presentation, UI, Services, Content.
4. Написать минимальный тест или ручной сценарий, который воспроизводит проблему.
5. Исправить код в слое-владельце состояния.
6. Прогнать связанный тест и smoke test.
7. Обновить отчет спринта и known issues.

## Правила для спринтов 2-8

- Каждая заявленная user story должна иметь хотя бы один тест-кейс.
- Каждая новая Warcraft II-механика должна ссылаться на строку `mechanics_matrix.md`.
- Если автоматический тест еще не готов, нужен ручной тест-кейс с четкими шагами.
- В отчете спринта нужно писать не "тесты есть", а какие именно проверки запускались.
- Неуспешные или заблокированные тесты не скрываются, а попадают в known issues.

## Минимальный набор для оценки спринта

Для каждого спринта в `docs/sprints/` должно быть:

- список реализованных user stories;
- список тест-кейсов;
- результат smoke test;
- список известных проблем;
- инструкция запуска демо;
- дата и время последней проверки.
